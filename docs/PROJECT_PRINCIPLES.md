# Принципы разработки — RegexHelper

**Единый источник глобальных правил проекта.** Используется при любых изменениях и в новых чатах — не полагаться только на промпты; сверяться с этим документом.

Общие правила при добавлении новых инструментов и изменении кода.

**Дата создания:** 2026-02-17

---

## Отдельные инструменты

- Каждый инструмент (Конвертер, Визуализатор, Тестер и т.д.) живёт в своей папке **tools/\<имя\>/**.
- **Не трогаем** модули других инструментов при разработке нового.
- Новые инструменты **разрабатываются отдельно** и **интегрируются** в текущий проект: добавляются в index.html, main.js, навигацию, подключаются к shared/core.

---

## Общие модули

- **core/** — конфиг, иконки, общие стили. Ядро, от которого зависят все инструменты.
- **shared/** — утилиты (storage, validation, escape), UI (notifications), контент (WIKI). Переиспользуются всеми инструментами.
- Инструменты **не меняют** core и shared без необходимости; добавляют свою логику в **tools/\<имя\>/**.

---

## Документация

- Документы инструмента хранятся в **docs/\<имя\>/** (docs/converter/, docs/visualizer/, и т.д.).
- Общие документы (STRUCTURE, DEPENDENCY_MAP, STATUS, PROJECT_PRINCIPLES) — в **docs/**.
- При добавлении инструмента: создать папку docs/\<имя\>/, SPECIFICATION, ARCHITECTURE и др.; обновить docs/STRUCTURE.md, docs/DEPENDENCY_MAP.md. **Новые файлы README не создавать** — важную информацию записывать в существующие документы по смыслу (SPECIFICATION, ARCHITECTURE, SETUP, TESTING и т.п.).
- **Единый README репозитория** в корне создаётся **только по явному указанию**, когда весь проект будет считаться полностью завершённым. До этого — не создавать README в корне и не использовать README как основной способ документирования (чтобы GitHub не показывал промежуточные README как главную страницу проекта).

---

## Глобальные правила (не терять)

- **Конвертер не трогать:** при разработке визуализатора, тестера и любых других инструментов модули и стили конвертера (**tools/converter/**) не изменять. Интеграция только через index.html, main.js, общие shared/core.
- **Не создавать README:** всю важную информацию фиксировать в текущей документации (docs/*, профильные .md в папках инструментов). Файлы README не создавать до момента, когда пользователь разрешит сделать один глобальный README по всему приложению.
- **Временные файлы:** папка **temp/** (или **tmp/**) в корне — для промптов, черновиков и прочего временного; содержимое можно удалять после использования, чтобы не засорять репозиторий.

---

## Терминал и команды (для агента / будущих чатов)

**Не запускать команды в терминале автоматически.** В среде пользователя автоматический запуск команд часто не срабатывает (ошибки оболочки, путь и т.д.).

- **Всегда присылать пользователю** точный текст команды (или список команд), что ввести, из какой папки запускать, и при необходимости — что сделать с выводом (прислать логи, скопировать ошибку и т.п.).
- **Пользователь сам** вводит команды в своём терминале и при необходимости отдаёт агенту вывод (логи, ошибки).
- Так мы избегаем ситуации, когда агент «запускает» команду, она падает из‑за окружения, а исправление (например, обрезка файла, сборка) не выполняется. Пользователь выполняет команду вручную — результат предсказуем.

**Пример:** вместо того чтобы вызывать `node temp/temp-trim.js` через инструмент терминала, написать: «Выполните из корня проекта: `node temp/temp-trim.js`. Пришлите вывод, если будет ошибка.»
